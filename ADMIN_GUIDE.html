What the user sees

  1. Admin creates a user in Supabase Dashboard and sends them a magic link or temporary password
  2. User clicks the magic link → browser stores the session token (localStorage)
  3. Every subsequent visit: app silently reads the token, confirms it with Supabase, and opens directly to the Welcome Screen
  4. Session stays valid for 1 hour (access token) but auto-refreshes silently in the background for up to 7 days (refresh token) — so users stay logged in without doing anything
  5. If session expires or is revoked → user sees a minimal "Access Required – Contact Admin" screen with no form

-- ── 1. Create tables if they don't exist yet ────────────

CREATE TABLE IF NOT EXISTS settings (
  user_id        UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  name           TEXT    DEFAULT '',
  rank           TEXT    DEFAULT '',
  "businessName" TEXT    DEFAULT 'MediAssist Pro',
  tagline        TEXT    DEFAULT 'Your Mobile Medical Companion',
  phone          TEXT    DEFAULT '',
  address        TEXT    DEFAULT '',
  photo          TEXT    DEFAULT '',
  logo           TEXT    DEFAULT ''
);

CREATE TABLE IF NOT EXISTS patients (
  id           TEXT PRIMARY KEY,
  user_id      UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  name         TEXT NOT NULL,
  age          INTEGER,
  gender       TEXT,
  phone        TEXT,
  address      TEXT,
  photo        TEXT,
  "createdAt"  TEXT,
  created_at   TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE IF NOT EXISTS visits (
  id             TEXT PRIMARY KEY,
  user_id        UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  "patientId"    TEXT NOT NULL,
  "patientName"  TEXT,
  date           TEXT,
  time           TEXT,
  notes          TEXT,
  services       JSONB    DEFAULT '[]',
  subtotal       NUMERIC  DEFAULT 0,
  discount       NUMERIC  DEFAULT 0,
  net            NUMERIC  DEFAULT 0,
  "createdAt"    TEXT,
  created_at     TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE IF NOT EXISTS services (
  id       TEXT PRIMARY KEY,
  user_id  UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  svc_id   INTEGER NOT NULL,
  name     TEXT NOT NULL,

CREATE TABLE IF NOT EXISTS hist_notes (
  id           TEXT PRIMARY KEY,
  user_id      UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  "patientId"  TEXT NOT NULL,
  date         TEXT,
  category     TEXT,
  title        TEXT,
  details      TEXT
);

-- ── 2. Indexes ────────────────────────────────────────────

CREATE INDEX IF NOT EXISTS idx_patients_user   ON patients   (user_id);
CREATE INDEX IF NOT EXISTS idx_visits_user     ON visits     (user_id);
CREATE INDEX IF NOT EXISTS idx_visits_patient  ON visits     ("patientId");
CREATE INDEX IF NOT EXISTS idx_services_user   ON services   (user_id);
CREATE INDEX IF NOT EXISTS idx_hist_user       ON hist_notes (user_id);
CREATE INDEX IF NOT EXISTS idx_hist_patient    ON hist_notes ("patientId");

-- ── 3. Enable RLS on all tables ──────────────────────────

ALTER TABLE settings   ENABLE ROW LEVEL SECURITY;
ALTER TABLE patients   ENABLE ROW LEVEL SECURITY;
ALTER TABLE visits     ENABLE ROW LEVEL SECURITY;
ALTER TABLE services   ENABLE ROW LEVEL SECURITY;
ALTER TABLE hist_notes ENABLE ROW LEVEL SECURITY;

-- ── 4. RLS policies (drop first so re-runs don't error) ───

DROP POLICY IF EXISTS "Users own their settings"    ON settings;
DROP POLICY IF EXISTS "Users own their patients"    ON patients;
DROP POLICY IF EXISTS "Users own their visits"      ON visits;
DROP POLICY IF EXISTS "Users own their services"    ON services;
DROP POLICY IF EXISTS "Users own their history notes" ON hist_notes;

CREATE POLICY "Users own their settings"
  ON settings FOR ALL
  USING      (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users own their patients"
  ON patients FOR ALL
  USING      (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users own their visits"
  ON visits FOR ALL
  USING      (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users own their services"
  ON services FOR ALL
  USING      (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users own their history notes"
  ON hist_notes FOR ALL
  USING      (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- ── 5. Auto-fill user_id trigger ─────────────────────────
-- Automatically sets user_id = auth.uid() on every INSERT.
-- Prevents any row from ever being written without an owner.
-- The frontend also passes user_id explicitly — this is a
-- safety net in case it's ever omitted.

CREATE OR REPLACE FUNCTION set_user_id()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.user_id IS NULL THEN
    NEW.user_id := auth.uid();
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

DROP TRIGGER IF EXISTS trg_patients_user_id   ON patients;
DROP TRIGGER IF EXISTS trg_visits_user_id     ON visits;
DROP TRIGGER IF EXISTS trg_services_user_id   ON services;
DROP TRIGGER IF EXISTS trg_hist_notes_user_id ON hist_notes;

CREATE TRIGGER trg_patients_user_id
  BEFORE INSERT ON patients
  FOR EACH ROW EXECUTE FUNCTION set_user_id();

CREATE TRIGGER trg_visits_user_id
  BEFORE INSERT ON visits
  FOR EACH ROW EXECUTE FUNCTION set_user_id();

CREATE TRIGGER trg_services_user_id
  BEFORE INSERT ON services
  FOR EACH ROW EXECUTE FUNCTION set_user_id();

CREATE TRIGGER trg_hist_notes_user_id
  BEFORE INSERT ON hist_notes
  FOR EACH ROW EXECUTE FUNCTION set_user_id();

-- ── 6. Enable Realtime for patients + visits ─────────────

ALTER PUBLICATION supabase_realtime ADD TABLE patients;
ALTER PUBLICATION supabase_realtime ADD TABLE visits;

You should see "Success. No rows returned". All tables, RLS policies, trigger, and realtime replication are now configured in one shot.

Create Users Manually (Admin Only)
The only way to create users — there is no signup UI in the app
Authentication → Users → Add user

Method A — Magic Link (Recommended)

  1. Go to Authentication → Users
  2. Click Add user → Create new user
  3. Enter the user's email address
  4. Leave password blank — click Create user
  5. Now click on that user row → click "Send magic link"
  6. User receives an email with a one-click login link
  7. They click it → browser opens the app → session is stored → they stay logged in

Method B — Set a Password

  1. Create the user the same way (Authentication → Users → Add user)
  2. After creating, click the user row
  3. Click "Send password reset" — user receives an email to set their own password
  4. Once they set a password, they can sign in using the Supabase-hosted auth page at https://YOUR_PROJECT_ID.supabase.co/auth/v1/...
⚠️
Magic Link is the cleanest method for this setup since there's no login form in the app. The user clicks the link once, the session is stored, and the app opens automatically every time after that — no clicking anything again unless the session expires.

What happens after the user's first login

  1. Supabase stores a session token in the user's browser localStorage
  2. Every time they open the app, getSession() reads this token silently
  3. If valid → bootApp() runs → Welcome Screen → Enter App → Dashboard
  4. Token auto-refreshes in the background — user stays logged in for 7 days of inactivity

5
Set Redirect URL
Magic links won't work without this
Authentication → URL Configuration
  1. Set Site URL to your app's URL, e.g. http://localhost:5500 for local testing or https://yourapp.netlify.app for production
  2. Under Redirect URLs, add:
    • http://localhost:5500
    • http://127.0.0.1:5500
    • Your production URL when deployed
  3. Click Save
ℹ️
When a user clicks a magic link, Supabase redirects them to this URL with the session token as a URL hash. The app reads it on load and stores the session. If the Site URL is wrong, magic links will redirect to the wrong place.

8
Session Lifespan & What Happens When It Expires
Understanding token lifetimes
Token typeDefault lifetimeBehaviour
Access token1 hourUsed for all API requests. Auto-refreshed silently in background.
Refresh token7 days (inactivity)Used to get new access tokens. Resets on every use.

In practice, as long as a user opens the app at least once every 7 days, they stay permanently logged in — the tokens keep refreshing. If they don't open it for 7+ days, the refresh token expires and they'll see the "Access Required" screen. You just send them a new magic link.

To extend session lifetime

Authentication → Providers → Email → expand

Increase "JWT expiry" (access token) and "Refresh token expiry" to your preferred values. For a daily-use medical app, 7-day JWT + 30-day refresh is reasonable.
